# 🔒 دليل قفل الفترات المحاسبية - Period Lock Guide

## 📋 نظرة عامة

**قفل الفترات المحاسبية** هي ميزة أمان محاسبي تمنع التعديل أو الحذف للقيود المحاسبية بعد إقفال فترة معينة (شهر محدد).

---

## 🎯 الهدف

### لماذا نحتاج لقفل الفترات؟

1. **الأمان المحاسبي**: منع التلاعب بالقيود القديمة
2. **الموثوقية**: ضمان صحة التقارير المالية
3. **الامتثال**: تلبية المتطلبات المحاسبية والقانونية
4. **التدقيق**: تسهيل عملية المراجعة والتدقيق

---

## 💻 التكامل مع الوحدات الأخرى

### 1. في وحدة السندات (Vouchers)

```javascript
// قبل حفظ سند جديد
const entryDate = new Date(document.getElementById('entryDate').value);

// التحقق من أن التاريخ ليس في فترة مقفلة
const isValid = await PeriodLockModule.validateEntryDate(entryDate);

if (!isValid) {
    // سيظهر تنبيه تلقائي: "الفترة مقفلة"
    return;
}

// إذا كان التاريخ صالح، تابع الحفظ
await saveVoucher(voucherData);
```

---

### 2. في وحدة القيود اليومية (General Entries)

```javascript
// قبل تعديل قيد موجود
async function editEntry(entryId) {
    const entry = await getEntryById(entryId);
    
    // التحقق من أن تاريخ القيد ليس في فترة مقفلة
    const isValid = await PeriodLockModule.validateEntryDate(entry.date);
    
    if (!isValid) {
        return; // لا يمكن التعديل
    }
    
    // إذا كان صالح، افتح نافذة التعديل
    showEditModal(entry);
}
```

---

### 3. في وحدة المبيعات/المشتريات

```javascript
// قبل حفظ فاتورة
async function saveInvoice(invoiceData) {
    // التحقق من تاريخ الفاتورة
    const isValid = await PeriodLockModule.validateEntryDate(invoiceData.date);
    
    if (!isValid) {
        return;
    }
    
    // حفظ الفاتورة
    await db.collection('salesInvoices').add(invoiceData);
}
```

## 🔑 الدوال الرئيسية

### 1. `loadCurrentPeriod()`
تحميل الفترة المحاسبية الحالية (الشهر والسنة الحالية).

---

### 2. `lockCurrentPeriod()`
قفل الفترة المحاسبية الحالية.

**الإجراءات:**
1. التحقق من أن الفترة ليست مقفلة بالفعل
2. حساب عدد القيود في هذه الفترة
3. الحصول على معلومات المستخدم الحالي
4. تحديث حالة الفترة إلى "locked"
5. حفظ معلومات القفل (التاريخ والمستخدم)

---

### 3. `lockPeriod(periodId)`
قفل فترة محددة بواسطة ID.

---

### 4. `unlockPeriod(periodId)`
فتح فترة مقفلة (يحتاج صلاحيات).

---

### 5. `isDateLocked(date)`
التحقق من أن تاريخ معين في فترة مقفلة.

**الإرجاع:**
- `true` إذا كان التاريخ في فترة مقفلة
- `false` إذا كان التاريخ في فترة مفتوحة

**الاستخدام:**
```javascript
const date = new Date('2025-09-15');
const isLocked = await PeriodLockModule.isDateLocked(date);

if (isLocked) {
    console.log('لا يمكن التعديل - الفترة مقفلة');
}
```

---

### 6. `validateEntryDate(date)` ⭐ (الأهم)
التحقق من صلاحية تاريخ قيد للإضافة/التعديل.

**الإرجاع:**
- `true` إذا كان التاريخ صالح (فترة مفتوحة)
- `false` إذا كان التاريخ غير صالح (فترة مقفلة) + يعرض تنبيه تلقائي

**الاستخدام في جميع الوحدات:**
```javascript
// قبل حفظ أي قيد محاسبي
const isValid = await PeriodLockModule.validateEntryDate(entryDate);

if (!isValid) {
    return; // أوقف العملية
}

// تابع الحفظ
```

## ⚠️ ملاحظات مهمة

### 1. **الفترة تُنشأ تلقائياً**
عند أول دخول إلى وحدة قفل الفترات، إذا لم تكن الفترة الحالية موجودة، سيتم إنشاؤها تلقائياً بحالة "مفتوحة".

---

### 2. **عدد القيود يُحسب عند القفل**
عند قفل فترة، يقوم النظام بحساب عدد القيود المحاسبية في تلك الفترة تلقائياً.

---

### 3. **معلومات المستخدم تُحفظ**
عند قفل فترة، يتم حفظ:
- اسم المستخدم
- معرف المستخدم
- تاريخ ووقت القفل

---

### 4. **التحذيرات واضحة**
عند محاولة قفل فترة، يظهر تحذير واضح:
```
⚠️ تحذير: بعد القفل لن يمكن التعديل على أي قيود في هذه الفترة!
```

---

### 5. **الفتح يحتاج تأكيد**
عند فتح فترة مقفلة، يظهر تحذير:
```
⚠️ تحذير: بعد الفتح سيمكن التعديل على القيود في هذه الفترة!
```

4. تحديث حالة الفترة إلى "locked"
5. حفظ معلومات القفل (التاريخ والمستخدم)

---

### 3. `lockPeriod(periodId)`
قفل فترة محددة بواسطة ID.

---

### 4. `unlockPeriod(periodId)`
فتح فترة مقفلة (يحتاج صلاحيات).

---

### 5. `isDateLocked(date)`
التحقق من أن تاريخ معين في فترة مقفلة.

**الإرجاع:**
- `true` إذا كان التاريخ في فترة مقفلة
- `false` إذا كان التاريخ في فترة مفتوحة

**الاستخدام:**
```javascript
const date = new Date('2025-09-15');
const isLocked = await PeriodLockModule.isDateLocked(date);

if (isLocked) {
    console.log('لا يمكن التعديل - الفترة مقفلة');
}
```

---

### 6. `validateEntryDate(date)` ⭐ (الأهم)
التحقق من صلاحية تاريخ قيد للإضافة/التعديل.

**الإرجاع:**
- `true` إذا كان التاريخ صالح (فترة مفتوحة)
- `false` إذا كان التاريخ غير صالح (فترة مقفلة) + يعرض تنبيه تلقائي

**الاستخدام في جميع الوحدات:**
```javascript
// قبل حفظ أي قيد محاسبي
const isValid = await PeriodLockModule.validateEntryDate(entryDate);

if (!isValid) {
    return; // أوقف العملية
}

// تابع الحفظ
```

## ⚠️ ملاحظات مهمة

### 1. **الفترة تُنشأ تلقائياً**
عند أول دخول إلى وحدة قفل الفترات، إذا لم تكن الفترة الحالية موجودة، سيتم إنشاؤها تلقائياً بحالة "مفتوحة".

---

### 2. **عدد القيود يُحسب عند القفل**
عند قفل فترة، يقوم النظام بحساب عدد القيود المحاسبية في تلك الفترة تلقائياً.

---

### 3. **معلومات المستخدم تُحفظ**
عند قفل فترة، يتم حفظ:
- اسم المستخدم
- معرف المستخدم
- تاريخ ووقت القفل

---

### 4. **التحذيرات واضحة**
عند محاولة قفل فترة، يظهر تحذير واضح:
```
⚠️ تحذير: بعد القفل لن يمكن التعديل على أي قيود في هذه الفترة!
```

---

### 5. **الفتح يحتاج تأكيد**
عند فتح فترة مقفلة، يظهر تحذير:
```
⚠️ تحذير: بعد الفتح سيمكن التعديل على القيود في هذه الفترة!
```
